# Разбор алгоритмов проекта

Этот документ описывает, как в проекте проходят данные от Excel-файла до
готового расписания, проверки конфликтов и экспорта в календарь.

## Общая схема работы

Основной сценарий состоит из нескольких этапов:

1. Пользователь вводит данные вручную на сайте или загружает Excel-файл.
2. Данные приводятся к JSON-структуре с аудиториями, преподавателями,
   предметами и группами.
3. Планировщик разворачивает учебную нагрузку в список отдельных занятий и
   подбирает для каждого занятия время, преподавателя и аудиторию.
4. Проверяющий модуль ищет конфликты в готовом расписании.
5. Готовое расписание можно выгрузить в Excel, PDF или календарь ICS.

В проекте используются две JSON-структуры:

```json
{
  "аудитории": [],
  "преподаватели": [],
  "предметы": [],
  "группы": []
}
```

Эта структура нужна для генерации расписания.

```json
{
  "lessons": [],
  "unplaced_lessons": []
}
```

Эта структура хранит результат генерации. В `lessons` лежат успешно
размещённые занятия, а в `unplaced_lessons` занятия, для которых не нашлось
допустимого места.

## Конвертация Excel в JSON

Файл `excel_to_json.py` подготавливает данные для остальных алгоритмов.

### Какие Excel-файлы поддерживаются

Конвертер распознаёт два типа книг:

1. Книгу с исходными данными для генерации.
2. Книгу с уже готовым расписанием.

Если в книге есть листы с аудиториями, преподавателями, предметами и группами,
она считается книгой исходных данных. Если таких листов нет, конвертер ищет
лист готового расписания: `Все занятия`, `schedule` или `lessons`.

### Нормализация листов и колонок

Названия листов и заголовков приводятся к единому виду:

- убираются лишние пробелы;
- регистр не влияет на распознавание;
- `ё` и `е` считаются одинаковыми;
- поддерживаются русские и английские алиасы некоторых названий.

Например, лист `auditoriums` и лист `аудитории` попадают в один раздел
`аудитории`.

### Разбор строк

Первая строка листа считается строкой заголовков. Остальные строки
превращаются в словари. Полностью пустые строки пропускаются.

Конвертер отдельно обрабатывает разные типы данных:

- обязательные числовые поля проверяются через `parse_int`;
- необязательные числа получают значение по умолчанию;
- списки id и условий разбираются из строки через запятую или точку с
  запятой;
- тип занятия приводится к коду:
  `1` для лекции и `0` для семинара.

Если в строке нельзя прочитать обязательное поле, ошибка содержит номер строки
Excel и имя проблемного поля. Это помогает исправить шаблон до запуска
планировщика.

### Результат конвертации исходных данных

Каждый лист превращается в свой список:

- аудитории содержат id, номер, вместимость и характеристики;
- преподаватели содержат id, ФИО, недельную нагрузку и id предметов;
- предметы содержат id, название, часы, тип занятия, длительность и условия;
- группы содержат id, номер, размер и id предметов.

Именно эту структуру читает функция `schedule` из `scheduler.py`.

### Результат конвертации готового расписания

Если конвертер читает лист готового расписания, каждая строка становится
элементом `lessons` со следующими полями:

- день недели;
- время;
- группа;
- предмет;
- тип;
- преподаватель;
- аудитория.

Список `unplaced_lessons` в этом сценарии создаётся пустым, потому что книга
уже содержит готовое расписание, а не исходные задачи для генерации.

## Генерация демонстрационных данных

Файл `schedule_data_fake.py` создаёт случайный Excel-файл `schedule_data.xlsx`.
Он нужен не для построения самого расписания, а для подготовки тестовой или
демонстрационной базы входных данных.

### Параметры объёма данных

В начале файла задаются количества создаваемых сущностей:

```python
n_auditoriums = 10
n_teachers = 10
n_subject = 5
n_groups = 5
```

Эти значения определяют размер сгенерированной базы. При каждом запуске
результат меняется, потому что используются функции модуля `random` и генератор
имён `Faker`.

### Генерация аудиторий

Сначала формируется список `auditoriums`.

Для каждой аудитории:

1. id берётся из номера итерации;
2. номер кабинета строится как `100 + id`;
3. вместимость выбирается случайно в диапазоне от 30 до 120;
4. набор характеристик формируется случайно.

Признак `тех` добавляется с вероятностью 20 %, а признак `экран` с
вероятностью 50 %. Если ни один признак не выпал, в список характеристик
добавляется `-`.

В итоге каждая запись аудитории имеет вид:

```python
{
    "id": i,
    "номер": 100 + i,
    "вместимость": random.randint(30, 120),
    "тип аудитории": type
}
```

### Генерация предметов

Предметы берутся из заранее заданного списка названий. Для первых
`n_subject` названий создаются две записи:

- лекция с типом `1`;
- семинар с типом `0`.

Поэтому при `n_subject = 5` итоговый список `subject` содержит 10 записей:
по две записи на каждое название предмета.

Для каждой записи случайно выбираются:

- часы в неделю из значений `2`, `4` или `6`;
- длительность занятия: 45 или 90 минут;
- дополнительные требования к аудитории.

Требования строятся так же, как характеристики аудитории:

- `тех` с вероятностью 20 %;
- `экран` с вероятностью 50 %;
- `-`, если дополнительных условий нет.

После генерации создаются:

- `subject_ids` - список id всех записей предметов;
- `subject_dict` - словарь `id -> название`.

В текущем коде `subject_dict` дальше не используется.

### Генерация преподавателей

Имена преподавателей формируются через:

```python
fake = Faker("ru_RU")
fake.name()
```

Так в тестовой базе появляются случайные русскоязычные ФИО.

Для распределения предметов используется словарь `sub_to_teach`. Он хранит,
какие преподаватели уже назначены на предмет.

Для каждого преподавателя алгоритм:

1. выбирает желаемое количество предметов от 1 до 3;
2. формирует список предметов, у которых пока меньше трёх преподавателей;
3. случайно выбирает допустимый набор;
4. записывает id преподавателя в `sub_to_teach`;
5. сохраняет преподавателя с недельной нагрузкой от 12 до 30 часов.

После основного прохода выполняется страховочный проход. Если какой-то предмет
не получил ни одного преподавателя, он добавляется случайному преподавателю.

### Генерация групп

Список `groups` формируется похожим способом.

Для каждой группы:

1. задаётся id;
2. номер строится как `10 + id`;
3. количество студентов выбирается случайно от 10 до 30;
4. выбирается список предметов.

Словарь `sub_to_gr` контролирует распределение предметов между группами.
Алгоритм старается выбирать предметы, которые пока назначены менее чем трём
группам.

После основного прохода выполняется дополнительная проверка: если какой-то
предмет не попал ни в одну группу, он добавляется случайной группе.

### Сборка структуры данных

После создания списков они объединяются в словарь:

```python
data = {
    "аудитории": auditoriums,
    "преподаватели": teachers,
    "предметы": subject,
    "группы": groups
}
```

Эти ключи становятся названиями листов Excel-книги.

### Запись Excel-файла

Для записи используется `openpyxl`:

1. создаётся новая книга `Workbook`;
2. стандартный пустой лист удаляется;
3. для каждого раздела `data` создаётся отдельный лист;
4. ключи первой записи раздела становятся заголовками колонок;
5. каждая запись превращается в строку листа;
6. файл сохраняется под именем `schedule_data.xlsx`.

Списки характеристик аудитории и условий предмета записываются как строки через
запятую.

### Особенность ссылок на предметы

В данных преподавателей и групп предметы сначала хранятся как id. Но при записи
в Excel скрипт заменяет их на читаемые подписи:

```text
математика (лекция), математика (семинар)
```

Это делает книгу понятнее человеку. Конвертер `excel_to_json.py` строит
справочник предметов и переводит такие подписи обратно в числовые id. Поэтому
он поддерживает оба формата ссылок на предметы:

- числовые id из шаблонной Excel-книги;
- подписи с названием и типом занятия из `schedule_data_fake.py`.

### Особенности выбранных id предметов

Список `subject` содержит отдельные записи для лекций и семинаров, поэтому его
id идут от 1 до `2 * n_subject`. При этом распределение предметов между
преподавателями и группами в текущем коде выбирается по диапазону
`1..n_subject`.

При `n_subject = 5` это означает, что назначения активно выбираются только из
первых пяти записей списка предметов, хотя всего сгенерировано десять записей.
Оставшиеся записи сохраняются в листе предметов, но не получают обычного
распределения на этапе выбора преподавателей и групп.

### Результат работы

Скрипт не возвращает Python-объект наружу и не создаёт расписание. Его
результат - Excel-файл:

```text
schedule_data.xlsx
```

После сохранения он выводит сообщение:

```text
Excel база данных сохранена
```

## Генерация расписания

Основной алгоритм построения расписания находится в `scheduler.py`.

### Ограничения планировщика

Алгоритм работает в пределах заданного учебного дня:

- дни: понедельник - пятница;
- начало дня: `09:00`;
- конец дня: `18:00`;
- шаг перебора времени: 15 минут;
- академический час: 45 минут.

При размещении занятия учитываются следующие ограничения:

- одна группа не может одновременно посещать два занятия;
- один преподаватель не может вести два занятия одновременно;
- одна аудитория не может использоваться двумя занятиями одновременно;
- преподаватель должен быть связан с предметом;
- недельная нагрузка преподавателя не должна превышать указанное значение;
- вместимость аудитории должна покрывать размер всех групп занятия;
- аудитория должна быть свободна в выбранный интервал.

### Построение задач занятий

Сначала функция `build_lesson_tasks` переводит абстрактную учебную нагрузку в
конкретные задачи размещения.

Для каждого предмета вычисляется общее количество минут в неделю:

```text
часы в неделю * длительность академического часа
```

Затем эта нагрузка делится на длительность одного занятия. Если деление
нецелое, используется округление вверх. Поэтому даже остаток нагрузки создаёт
ещё одну задачу занятия.

Семинары и лекции разворачиваются по-разному:

- семинар создаётся отдельно для каждой группы;
- лекция одного предмета объединяет все группы, которым она нужна.

Из-за этого лекция может получить несколько групп в одном занятии, а её
требуемая вместимость аудитории равна сумме размеров этих групп.

### Порядок размещения задач

После создания задач они сортируются. Сначала планировщик пытается разместить:

1. более длинные занятия;
2. занятия предметов с большей недельной нагрузкой.

Это уменьшает риск, что длинное занятие останется без подходящего свободного
интервала после заполнения расписания короткими занятиями.

### Перебор вариантов

Для каждой задачи алгоритм перебирает:

1. день недели;
2. возможное время начала;
3. свободного преподавателя;
4. свободную аудиторию.

Сначала проверяется доступность всех групп. Если хотя бы одна группа уже занята
в этом интервале, вариант сразу исключается.

Затем выбирается преподаватель:

- он должен вести нужный предмет;
- он должен быть свободен;
- после добавления занятия его недельная нагрузка не должна превысить лимит.

После этого выбирается аудитория:

- она должна быть свободна;
- она должна вмещать всех студентов;
- она сортируется по пригодности к условиям предмета.

### Приоритет аудитории

Функция `aud_prior` оценивает, насколько аудитория подходит предмету.

Если предмет требует техническую аудиторию, лучший вариант - аудитория с
признаком `тех`. Аудитория с экраном считается менее подходящей, но всё ещё
лучше обычной. Аналогично работает требование `экран`.

Для лекций при равном приоритете предпочитается более вместительная аудитория.
Для семинаров предпочитается аудитория меньшей подходящей вместимости, чтобы не
занимать большой кабинет без необходимости.

### Оценка допустимого варианта

Если вариант не нарушает жёсткие ограничения, он получает числовую оценку:

```text
score =
    окна групп * 10
    + окна преподавателя * 5
    + приоритет аудитории * 100
    + близость ко времени начала дня * 0.01
    + текущая нагрузка преподавателя
```

Чем меньше `score`, тем лучше вариант.

Из формулы видно, что:

- неподходящая аудитория штрафуется сильнее всего;
- алгоритм старается уменьшать окна у групп;
- окна преподавателя тоже нежелательны, но имеют меньший вес;
- более ранние занятия имеют небольшой приоритет;
- преподаватели с меньшей текущей нагрузкой получают небольшой приоритет.

### Фиксация занятия

Когда найден лучший вариант, алгоритм:

1. сохраняет занятость каждой группы;
2. сохраняет занятость преподавателя;
3. сохраняет занятость аудитории;
4. увеличивает использованную нагрузку преподавателя;
5. добавляет занятие в `lessons`.

Если ни один вариант не подошёл, занятие добавляется в `unplaced_lessons`.
Там сохраняются группа, предмет, тип, длительность и общая причина:
не найден свободный преподаватель, кабинет или интервал времени с учётом
ограничений.

### Характер алгоритма

Планировщик является жадным. Он выбирает лучший вариант для текущей задачи,
фиксирует его и переходит к следующей. Он не делает полный перебор всех
расписаний и не откатывает ранее поставленные занятия, если позже появилась
сложная задача.

Плюс такого подхода - простая и быстрая генерация. Ограничение - результат
может быть не глобально оптимальным, особенно если входные данные сильно
плотные.

## Динамическое изменение расписания

Файл `dynamic_scheduler.py` работает не с исходной генерацией, а с уже
созданным расписанием.

### Подготовка данных

Функция `normalize_schedule` делает копию расписания и добавляет каждому
занятию `_id`, если его ещё нет. Эти id нужны интерфейсу, чтобы пользователь
мог выбрать конкретные занятия для изменения.

Функция `extract_options` собирает варианты для фильтров:

- преподавателей;
- предметы;
- аудитории;
- группы;
- дни;
- уже встречающиеся интервалы времени.

Часть вариантов берётся из исходных справочников, а часть из самого
расписания.

### Доступные изменения

Функция `apply_dynamic_change` поддерживает несколько действий:

- заменить преподавателя у выбранных занятий;
- удалить выбранные занятия;
- перенести выбранные занятия в свободный слот;
- заменить предмет, преподавателя или аудиторию у занятия;
- добавить окно в расписание группы.

### Поиск свободного слота

При переносе занятия используется `find_free_slot`.

Алгоритм берёт длительность исходного занятия, перебирает дни и интервалы с
шагом 15 минут и создаёт временный кандидат. Затем кандидат сравнивается со
всеми существующими занятиями.

Конфликт фиксируется, если одновременно совпали:

- день;
- пересекающийся временной интервал;
- хотя бы один ресурс: группа, преподаватель или аудитория.

Динамический модуль не подбирает нового преподавателя по нагрузке и не
пересчитывает вместимость аудитории. Он меняет готовое расписание точечно и
проверяет занятость тех значений, которые уже записаны в занятии.

## Проверка расписания

Файл `checker.py` проверяет готовое расписание после генерации или загрузки.

### Нормализация занятия

Каждое занятие проходит через `norm_lesson`:

1. строка времени переводится в минуты начала и конца;
2. поле `группа` разбивается по запятым;
3. день, преподаватель и аудитория очищаются от лишних пробелов;
4. собирается читаемое имя занятия для текста ошибки.

Если время или обязательные поля разобрать нельзя, в результат проверки
добавляется ошибка конкретного занятия, а проверка остальных строк продолжается.

### Индексы занятости

Проверяющий алгоритм строит три словаря:

- занятия по группам;
- занятия по преподавателям;
- занятия по аудиториям.

Одна лекция для нескольких групп попадёт в индекс каждой группы отдельно.
Благодаря этому модуль проверяет конфликты по каждому ресурсу.

### Проверка пересечений

Для каждого ресурса занятия сортируются по дню и времени. Затем сравниваются
пары занятий одного дня.

Два интервала пересекаются, если выполняется условие:

```text
start1 < end2 и start2 < end1
```

Если пересечение найдено:

- для группы формируется сообщение о двух занятиях в одно время;
- для преподавателя формируется сообщение о двойной занятости;
- для аудитории формируется сообщение о двойном использовании кабинета.

В ошибку попадает не весь интервал занятий, а их фактическое общее время.

### Что проверка не делает

Проверка не создаёт расписание заново и не исправляет ошибки. Она также не
считает `unplaced_lessons` конфликтами: эти занятия выводятся отдельно как
задачи, которые генератор не смог разместить.

## Конвертация JSON в ICS

Файл `json_to_calendar.py` экспортирует готовое расписание в формат календаря.

### Выбор недели

ICS-событию нужна точная календарная дата, а в расписании хранится только день
недели. Поэтому конвертер выбирает понедельник учебной недели:

- из параметра `week_start`, если он передан;
- как ближайший понедельник, если параметр не указан.

Название дня недели переводится в смещение от понедельника. Например,
понедельник имеет смещение 0, пятница - 4.

### Построение события

Каждое занятие превращается в блок `VEVENT`.

Из строки времени вычисляются:

- `DTSTART` - начало события;
- `DTEND` - конец события.

Из данных занятия заполняются:

- `SUMMARY` - короткое название;
- `DESCRIPTION` - предмет, тип, группа, преподаватель и аудитория;
- `LOCATION` - аудитория.

Если время окончания оказалось меньше или равно началу, алгоритм считает, что
событие заканчивается на следующем дне.

### UID и повторения

Для каждого события создаётся UID. Он вычисляется как SHA-1 от даты события и
основных полей занятия. Это делает идентификатор воспроизводимым для одного и
того же экспортируемого занятия.

Если параметр `repeat_weeks` больше 1, в событие добавляется правило:

```text
RRULE:FREQ=WEEKLY;COUNT=N
```

Так календарь может показать занятие не один раз, а повторять его несколько
недель.

### Соблюдение формата ICS

Перед записью:

- специальные символы экранируются;
- переносы текста превращаются в безопасные последовательности;
- строки длиннее 75 UTF-8 байт складываются по правилам iCalendar;
- в календарь записывается часовой пояс.

На выходе получается `.ics`-файл, который можно импортировать в календарное
приложение.

## Связь модулей

Кратко зависимости выглядят так:

```text
Excel с данными
    -> excel_to_json.py
    -> scheduler.py
    -> checker.py
    -> json_to_calendar.py
```

Для изменений после генерации используется дополнительный путь:

```text
готовое расписание
    -> dynamic_scheduler.py
    -> checker.py
    -> экспорт
```

Главная идея проекта в том, что генератор работает с исходными справочниками,
проверка работает с уже размещёнными занятиями, а конвертеры переводят данные
между внешними форматами и JSON-структурами проекта.
